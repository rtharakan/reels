## Testing Readiness — Scenario Coverage

- [x] CHK032 - Are requirements defined for the scenario where a user deletes their account while another user has them in their Discover feed or pending interest? [Coverage, Edge Case, Gap] → **Resolved**: New edge case added to spec — soft-deleted users excluded from queries; client-side stale cards return NOT_FOUND.
- [x] CHK033 - Are requirements defined for what happens when a user blocks someone they've already matched with — is the match deleted or hidden? [Coverage, Spec §FR-026, Data Model §Block] → **Resolved**: FR-026 updated: "Any existing Match and Interest records between the two users MUST be deleted upon blocking."
- [x] CHK034 - Are requirements specified for re-import behavior — does re-import replace the entire watchlist or merge incrementally? [Coverage, Spec §FR-014, Gap] → **Resolved**: New edge case added to spec: upsert behavior (add new, keep existing, remove absent), match scores recomputed.
- [x] CHK035 - Are requirements defined for the scenario where TMDB API is unreachable during watchlist import — partial import, retry, or fail-all? - [x] CHK036 - Are session expiry UX requirements defined — what happens when a JWT expires mid-action (e.g., during Discover swiping or form submission)? [Coverage, Spec §FR-004a, Gap] → **Resolved**: FR-004a updated: client silently refreshes; if refresh token expired, redirect to login.
- [x] CHK037 - Are requirements specified for handling duplicate Interest records (user taps "interest" twice rapidly)? The unique constraint handles it at DB level, but is the UX defined? [Coverage, Edge Case, Gap] → **Resolved**: New edge case added: unique constraint prevents duplicates, second request returns success idempotently.
- [x] CHK038 - Are requirements defined for what happens when a user's Letterboxd profile changes from public to private after initial import? [Coverage, Edge Case, Gap] → **Resolved**: New edge case added: re-import fails with error, existing data preserved, user remains matchable.
- [x] CHK039 - Are requirements defined for the empty Matches list state (user has no matches yet)? [Coverage, Spec §US3, Gap] → **Resolved**: New edge case added: empty state "No matches yet — keep discovering!" with no nudge.
- [x] CHK040 - Are recovery flow requirements defined for a failed onboarding (user closes browser mid-onboarding, returns later)? The middleware routes incomplete users back, but is partial data preserved? [Coverage, Edge Case, Gap] → **Resolved**: New edge case added: partial data NOT preserved — user restarts from step 1. This avoids storing profile data before consent (Privacy by Design).
